# Arcana Gateway Protocol (v2)

This document describes the Gateway control-plane protocol for Arcana v2.
It focuses on HTTP and WebSocket APIs, the core resource model, and how the
model maps onto the current implementation. The v2 Gateway is implemented
under `src/gateway-v2` and exposes `/v2/*` HTTP and WebSocket endpoints.
Arcana's chat HTTP stack is now served by Gateway v2 (see `src/gateway-v2`) for the desktop and CLI. Chat flows use the `/v2/*` HTTP endpoints plus the `/v2/stream` WebSocket exclusively; the old `/api/chat2` endpoint remains removed, and the legacy SSE stream `/api/events` is no longer used by the current web UI.

## Goals

- Provide a single control-plane entry point for all ingress paths
  (web UI, CLI, channel bridges, cron, heartbeat, services).
- Normalize how agents, sessions, events, traces, and state are modeled.
- Offer a WebSocket stream for low-latency event and trace delivery.
- Keep the design close to the existing SSE event stream so
  the Gateway can be implemented as a thin layer over the current code.

## Resource model

The Gateway exposes a small set of logical resources. These map onto the
existing multi-agent model described in `docs/multi-agent.md`.

### Agent

- Identity: `agentId` (string).
- Backing store: `~/.arcana/agents/<agentId>/agent.json`.
- Fields (logical view):
  - `id`: matches `agentId` from `agent.json` and `/api/agents`.
  - `workspaceRoot`: absolute path to the workspace.
  - `createdAt`: ISO timestamp.
  - `metadata`: opaque agent-level metadata (labels, description, etc.).

### Session

- Identity: `(agentId, sessionId)`.
- Backing store: `~/.arcana/agents/<agentId>/sessions/<sessionId>.json`.
- Fields (logical view):
  - `id`: session id.
  - `agentId`: owning agent.
  - `title`: human-friendly title.
  - `createdAt`, `updatedAt`.
  - `summary`: optional conversation summary (see cron compaction).
  - `tags`: optional labels (channel id, external thread id, etc.).

### Event

Events capture fine-grained activity across the system. Today, these are delivered over the Gateway v2 `/v2/stream` WebSocket and emitted on the process-local event bus in `src/event-bus.js`.

- Identity: `eventId` (string, generated by Gateway).
- Scope: either global, per agent, or per session.
- Common envelope fields:
  - `eventId`: unique id.
  - `type`: event kind (for example `assistant_text`).
  - `agentId` (optional for global events).
  - `sessionId` (optional for global/agent events).
  - `turnId` (optional, for events within a single turn).
  - `source`: `agent` | `tool` | `channel` | `cron` | `heartbeat` | `system`.
  - `ts`: ISO timestamp.
  - `traceId`, `spanId` (optional; see Trace below).
  - `data`: type-specific payload object.

Examples of event types (non-exhaustive, aligned with current SSE events):

- `server_info`.
- `turn_start`, `turn_end`.
- `thinking_start`, `thinking_progress`, `thinking_end`.
- `assistant_text`, `assistant_image`.
- `tool_execution_start`, `tool_execution_update`, `tool_execution_end`.
- `tool_repeat`, `memory_trigger`, `llm_usage`.
- `history_compact_start`, `history_compact_end`.
- `steer_enqueued`, `abort_done`, `skills_refresh`.

### Trace

Traces provide structured spans for observability. The current code already
emits usage and tool events; the Gateway formalizes these into a trace model.

- Identity: `traceId` (string).
- A trace contains one or more spans:

```jsonc
{
  "traceId": "trc-2026-03-08-abc",
  "rootSpanId": "span-1",
  "spans": [
    {
      "spanId": "span-1",
      "parentSpanId": null,
      "name": "session.turn",
      "startTime": "2026-03-08T12:34:56.000Z",
      "endTime": "2026-03-08T12:34:58.500Z",
      "status": "ok",
      "attributes": {
        "agentId": "default",
        "sessionId": "2026-03-04__...",
        "policy": "restricted"
      }
    },
    {
      "spanId": "span-2",
      "parentSpanId": "span-1",
      "name": "tool.execute",
      "startTime": "2026-03-08T12:34:56.900Z",
      "endTime": "2026-03-08T12:34:57.200Z",
      "status": "error",
      "attributes": {
        "tool": "web_search",
        "ok": false
      }
    }
  ]
}
```

The Gateway does not require a specific tracing backend; it only standardizes
the JSON representation and how spans are correlated with events.

### State

State represents a derived view over sessions and agents.

- `SessionState` (per `(agentId, sessionId)`):
  - `status`: `idle` | `running` | `aborted`.
  - `lastEventId`.
  - `sessionTokens`: total tokens consumed (matches `llm_usage`).
  - `lastTurn`: summary of the most recent turn (start/end times, policy).
  - `channel`: optional channel metadata (for example `feishu_group_<chat>`).

State can be queried via HTTP and also emitted periodically as `session_state`
events on the Gateway stream.

## HTTP control-plane API (Proposed)

All endpoints are versioned under `/v2/gateway` (logical). Today the concrete HTTP surface lives under `/v2/*` (see `src/gateway-v2/index.js`), and chat flows use `/v2/turn` + `/v2/turn-sync` directly.

### Create or load a session

`POST /v2/gateway/agents/:agentId/sessions`

Request body:

```jsonc
{
  "title": "New session",      // optional
  "sessionId": "2026-03-04__...", // optional; reuse when provided
  "tags": { "channel": "feishu_group_..." }
}
```

Response:

```jsonc
{
  "session": {
    "id": "2026-03-04__...",
    "agentId": "default",
    "title": "New session",
    "createdAt": "...",
    "updatedAt": "...",
    "tags": { "channel": "feishu_group_..." }
  }
}
```

Implementation note: this is a thin wrapper over `POST /api/sessions` and the
sessions store described in `docs/multi-agent.md`.

### Start a turn (chat invocation)

`POST /v2/gateway/sessions/:sessionId/turns`

Request body:

```jsonc
{
  "agentId": "default",          // required
  "message": "...",              // required user text
  "policy": "open" | "restricted", // optional, defaults to restricted
  "metadata": {
    "channelMessageId": "...",   // optional channel-specific ids
    "reason": "user_message"     // user_message | cron | heartbeat | service
  }
}
```

Response (non-streaming):

```jsonc
{
  "turnId": "turn-001",
  "sessionId": "2026-03-04__...",
  "text": "assistant reply text"
}
```

Implementation note: in the current codebase this is implemented by Gateway v2
endpoints (see `src/gateway-v2/index.js` and `reactor-runner.js`) and fully
replaces the legacy `POST /api/chat2` handler (which has been removed).

### Append external events

`POST /v2/gateway/sessions/:sessionId/events`

Used by channel runtimes, cron, and services to append structured events into a
session without necessarily starting a turn.

Request body:

```jsonc
{
  "agentId": "default",
  "events": [
    {
      "type": "channel.message",
      "source": "channel",
      "ts": "2026-03-08T12:34:56.123Z",
      "data": {
        "channel": "feishu_group_<chatId>",
        "text": "raw inbound text",
        "dedupeKey": "feishu:<msgId>"
      }
    }
  ]
}
```

The Gateway persists these as `Event` records and may apply ingress policy
before deciding whether to enqueue a new turn.

### Read session state

`GET /v2/gateway/sessions/:sessionId/state?agentId=default`

Returns the current `SessionState` derived from recent events and traces.

## WebSocket event stream

The WebSocket endpoint replaces the SSE endpoint `/api/events` for the chat UI.
The current implementation streams all events on `/v2/stream` without an explicit
subscribe handshake; the subscribe message shape below is the long-term design.

- Endpoint: `GET ws://<host>/v2/stream` (or `wss://` in production).
- Messages are JSON objects per frame.

### Client subscribe message (design)

In the full design, clients send a subscribe request after connecting:

```jsonc
{
  "type": "subscribe",
  "agentId": "default",          // optional; default agent when omitted
  "sessionId": "2026-03-04__...", // optional; subscribe to all when omitted
  "events": ["assistant_text", "tool_execution_*"],
  "includeTraces": true,
  "includeState": true
}
```

### Event delivery message

Events delivered over the WebSocket reuse the `Event` envelope:

```jsonc
{
  "type": "assistant_text",
  "eventId": "evt-2026-03-08-001",
  "agentId": "default",
  "sessionId": "2026-03-04__...",
  "turnId": "turn-001",
  "source": "agent",
  "ts": "2026-03-08T12:34:57.000Z",
  "data": { "text": "assistant reply text" },
  "traceId": "trc-2026-03-08-abc",
  "spanId": "span-1"
}
```

Trace objects can be delivered either embedded in `data` for specific event
types (for example `tool_execution_end`) or as separate `type: "trace"`
messages carrying the full trace document.

## Mapping to current implementation

This section connects the proposed Gateway model to existing code. All mappings
are descriptive; they do not imply that the Gateway endpoints exist yet.

### Gateway v2 chat (src/gateway-v2)

- Synchronous chat requests use `/v2/turn-sync` with `{ text, policy, agentId, sessionKey }`.
- Asynchronous chat requests use `/v2/turn` plus the `/v2/stream` WebSocket.
- The default runner (`reactor`) lives in `src/gateway-v2/reactor-runner.js` and
  uses `runArcanaTask` to execute turns.

### Event bus and Gateway v2 stream

- `src/event-bus.js` exposes `eventBus` and `emit(ev)`. Gateway v2 attaches to this bus and forwards events over `/v2/stream` as JSON envelopes with a `type` field.
- The legacy SSE endpoint `GET /api/events` from earlier versions created a long-lived SSE connection and forwarded events produced by sessions and the event bus. It has been removed in favor of the Gateway v2 WebSocket stream in current builds.
- The proposed Gateway WebSocket endpoint reuses the same event types and
  payloads, but standardizes the envelope (`eventId`, `agentId`, `sessionId`,
  `traceId`, `spanId`) and moves transport from SSE to WebSocket.

### Cron scheduler (`src/cron/*` and `docs/cron.md`)

- Cron jobs today use `runArcanaTask` (`src/cron/arcana-task.js`) to run
  background turns and write results into sessions. They also send Feishu
  replies in some cases.
- In the Gateway model, cron is conceptually an ingress source that appends
  `source: "cron"` events (for example `cron.job_started`, `cron.job_finished`)
  and triggers turns. In the current implementation, cron runs are wired
  directly through `runArcanaTask` and share the same event bus + state as
  Gateway chat turns.
- Usage metrics already emitted as `llm_usage` events can be wrapped into
  trace spans with `traceId`/`spanId` linked back to the originating job.

### Channel runtime (`docs/channel-runtime.md`)

- The current Channel Runtime describes how channel adapters normalize inbound
  messages and call Gateway v2 chat (`/v2/turn` or `/v2/turn-sync`). The legacy
  `/api/steer` endpoint remains available only for backwards compatibility and is
  no longer used by new channel bridges (they send a new turn instead of in-flight steering).
- In the Gateway model, a `ChannelPlugin` or adapter would:
  - Normalize inbound events to the `Event` schema.
  - Append them via `POST /v2/gateway/sessions/:sessionId/events`.
  - Start turns via `POST /v2/gateway/sessions/:sessionId/turns` when ingress
    policy decides a run should be triggered.
  - Listen on the WebSocket event stream for `assistant_text`, `assistant_image`,
    and `session_state` events to drive replies back to the channel.
- This keeps channel code focused on normalization and reply sending, while the
  Gateway owns sessions, turns, and observability.
